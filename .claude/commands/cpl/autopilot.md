# /cpl:autopilot

**Recommended model: opus**

Ты — автономный оркестратор конвейера. Получаешь цель и ведёшь пайплайн до её достижения: сам определяешь следующую фазу, сам выбираешь исполнителя (Codex или Claude-агент), сам проверяешь результат и продолжаешь цикл.

## Цель

$ARGUMENTS

---

## Шаг 0: Инициализация

1. Прочитай все существующие файлы в `ai/` (context.md, requirements.md, architecture.md, tasks.md, status.md и т.д.)
2. Определи **стартовую точку** — первый пропущенный артефакт в цепочке:
   ```
   context → requirements → architecture → tasks → [impl] → review → verify
   ```
3. Определи **финальную точку** исходя из цели в $ARGUMENTS (см. таблицу ниже)
4. **Обнаружь доступные MCP-серверы** — проверь, какие из известных серверов активны в текущей сессии, и сопоставь с каталогом:

   | MCP сервер | Признак наличия | Усиливает фазы |
   |------------|-----------------|----------------|
   | Figma | инструменты `mcp__claude_ai_Figma__*` | `design`, `ui-spec` — get_design_context, get_screenshot |
   | Pencil | инструменты `mcp__pencil__*` | `design`, `ui-spec` — batch_get, batch_design |
   | context7 | инструменты `mcp__plugin_context7_*` | `context` — документация незнакомых библиотек |
   | google-dev-knowledge | инструменты `mcp__google-dev-knowledge__*` | `context`, `investigate` — поиск по документации |
   | Telegram | инструменты `mcp__plugin_telegram_telegram__*` | уведомление о завершении (не фаза) |

   Если MCP-сервер из каталога активен — он становится **приоритетным исполнителем** для своих фаз.
   Неизвестные MCP-серверы игнорируй — не гадай об их назначении.

5. Создай `ai/autopilot.md`:
   ```markdown
   # Autopilot
   Цель: <цель из $ARGUMENTS>
   Старт: <стартовая фаза>
   Финиш: <финальная фаза>
   Итерация: 0
   Статус: running

   ## MCP Extensions
   - <mcp-сервер>: <фазы> (или "нет" если ни один не найден)

   ## Лог
   ```
6. Лимит: **максимум 10 итераций**. Если достигнут — СТОП и отчёт.

### Таблица финальных точек по цели

| Признаки цели | Финальная фаза |
|---------------|----------------|
| "контекст", "разберись", "посмотри" | `context` |
| "требования", "хочу фичу" | `requirements` |
| "архитектура", "спроектируй" | `architecture` |
| "реализуй", "напиши код", "сделай" | `verify` |
| "до ревью", "review-ready" | `code-review` |
| "полный цикл", "от нуля до PR" | `code-review` |
| "релиз", "задеплой" | `release` |

## Шаг N: Основной цикл

Повторяй шаги A–F пока не сработает одно из условий остановки.

### A. Определи следующую фазу

Прочитай `ai/status.md`. Применяй логику:
1. Если есть `⚠️ RE-PASS <фаза>` — это следующая фаза
2. Иначе — первый незаполненный артефакт в цепочке до финальной точки

### B. Маршрутизируй исполнителя

Приоритет: **MCP** (если обнаружен для этой фазы) → **Codex** → **Claude Agent**

| Фаза | Исполнитель по умолчанию | Если есть MCP |
|------|--------------------------|---------------|
| `context` / `gather` | **Codex** | context7 / google-dev-knowledge |
| `design` / `ui-spec` | **Claude Agent** | Figma или Pencil |
| `progress` | **Codex** | — |
| `changelog` | **Codex** | — |
| `verify` | **Codex** | — |
| `routine` | **Codex** | — |
| `requirements` / `product` | **Claude Agent** | — |
| `architecture` / `architect` | **Claude Agent** | — |
| `gen` / `gen-server` | **Claude Agent** | — |
| `code-review` | **Claude Agent** | — |
| `debug` / `investigate` | **Claude Agent** | google-dev-knowledge |
| `test-plan` / `gen-tests` | **Claude Agent** | — |
| `security` | **Claude Agent** | — |
| `refactor` | **Claude Agent** | — |

### C. Выполни фазу

#### Если MCP:

Используй соответствующий MCP-инструмент напрямую:
- **Figma/Pencil** для `design`/`ui-spec`: вызови `get_design_context` или `batch_get`, результат передай в Claude Agent для финального оформления артефакта
- **context7** для `context`: вызови `resolve-library-id` + `query-docs` для незнакомых зависимостей, включи вывод в `ai/context.md`
- **google-dev-knowledge** для `context`/`investigate`: вызови `answer_query`, включи в артефакт
- **Telegram**: не исполнитель фазы — используй только для уведомления при DONE/PAUSE/STOP

#### Если Codex:

1. Прочитай входные файлы фазы
2. Составь **самодостаточный промпт** (все нужные данные внутри)
3. Проверь наличие Codex:
   ```bash
   which codex
   ```
4. Если есть — запусти и сохрани exit code:
   ```bash
   codex exec --dangerously-bypass-approvals-and-sandbox "<промпт>"; echo "EXIT:$?"
   ```
   Для длинных промптов (>500 символов или многострочных) используй heredoc:
   ```bash
   codex exec --dangerously-bypass-approvals-and-sandbox "$(cat <<'EOF'
   <промпт>
   EOF
   )"; echo "EXIT:$?"
   ```
5. Проверь результат по exit code:
   - `EXIT:0` → фаза успешна, переходи к шагу D
   - `EXIT:<ненулевой>` → запиши ошибку в лог как `⚠️ RE-PASS`, повтори фазу (1 раз)
   - Второй ненулевой подряд → `❌ FAIL`, переходи к условию остановки
6. Если Codex не найден — выполни фазу сам (Claude), запиши в лог: `[fallback: codex not found]`

#### Если Claude Agent:

Запусти Agent с полным контекстом фазы. В промпте обязательно укажи:
- Текущую цель из autopilot
- Содержимое входных файлов фазы
- Какой артефакт нужно создать/обновить
- Ссылку на SKILL.md соответствующей фазы для инструкций

После возврата агента выполни три проверки по порядку:

1. **Артефакт существует**: попробуй прочитать ожидаемый файл (`ai/<фаза>.md` или иной по таблице фаз).
   - Файл не найден → `⚠️ RE-PASS`, повтори фазу (1 раз); второй раз — `❌ FAIL`
2. **Ответ агента не содержит маркеров провала**: ищи фразы «не могу», «нужно уточнить», «отсутствует», «недостаточно данных», «требуется».
   - Найдено → `⚠️ RE-PASS` с причиной из ответа
3. **Содержимое не пустое и не шаблонное**: артефакт должен содержать конкретику, а не только заголовки/плейсхолдеры.
   - Шаблонный → `⚠️ RE-PASS`; второй раз подряд → `❌ FAIL`

### D. Проверь результат

После завершения фазы (любым исполнителем):
1. Применяй проверки исполнителя (см. выше — exit code для Codex, три проверки для Agent)
2. Проверь на Level 3 deviation (см. `references/deviation-rules.md`):
   - новая таблица БД, новый сервис, смена паттерна, внешняя зависимость → **PAUSE**
3. Проверь на `⚠️ RE-PASS` в выводе артефакта → учти в следующей итерации
4. Для фазы `verify`: ненулевой exit code или маркер провала в ответе — задача **не выполнена**, `⚠️ RE-PASS verify`, не переходи к DONE

### E. Обнови лог

Дополни `ai/autopilot.md`:
```markdown
### Итерация N
- Фаза: `<фаза>`
- Исполнитель: MCP (<сервер>) | Codex | Claude Agent
- Результат: ✅ OK | ⚠️ RE-PASS | ❌ FAIL | ⏸️ PAUSE
- Артефакт: `ai/<файл>.md`
- Заметки: <если есть>
```

### F. Проверь условия остановки

| Условие | Действие |
|---------|----------|
| Финальная фаза выполнена без RE-PASS | **DONE** → финальный отчёт |
| Level 3 deviation обнаружен | **PAUSE** → спроси пользователя |
| Фаза провалилась дважды подряд | **STOP** → отчёт о блокере |
| Итерация = 10 | **STOP** → отчёт о прогрессе |
| Цель достигнута (артефакты заполнены) | **DONE** → финальный отчёт |

## Финальный отчёт

```
## Autopilot завершён

**Цель**: <цель>
**Статус**: ✅ DONE | ⏸️ PAUSE | ⛔ STOP

**Выполнено** (N итераций):
- Итерация 1: context → Codex ✅
- Итерация 2: requirements → Claude Agent ✅
- ...

**Артефакты**: ai/context.md, ai/requirements.md, ...

**Следующий шаг**: <если PAUSE/STOP — что нужно от пользователя>
```

Обнови `ai/autopilot.md` финальным статусом.

## Правила

- **НЕ пропускай фазы** — каждая фаза пишет артефакт, следующая его читает
- **НЕ изменяй бизнес-логику** при routing-правках — только механика
- **ВСЕГДА останавливайся** при Level 3 deviation, не угадывай решение
- **Codex-промпт самодостаточен** — локальный CLI не читает файловую систему автоматически
- **Один артефакт за итерацию** — не пытайся сделать несколько фаз в одном шаге
- Если пользователь не указал финальную точку — используй `code-review` по умолчанию

## Антипаттерны

- ❌ Запускать `gen` без `tasks.md` — нарушение пайплайна
- ❌ Игнорировать RE-PASS блокеры — цикл зайдёт в тупик
- ❌ Делать 10 итераций на одной фазе — признак бесконечного цикла, СТОП
- ❌ Передавать в Codex промпт без данных — получишь галлюцинации вместо работы
- ❌ Пропускать проверку Level 3 deviation — можешь сломать архитектуру незаметно
