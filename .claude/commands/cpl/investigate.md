# /cpl:investigate

**Recommended model: sonnet**

## Запрос
$ARGUMENTS

## Выполнение через Agent

**СТОП. Не выполняй задание — вызови Agent tool прямо сейчас.**
- `prompt`: весь текст этого скилла (включая `$ARGUMENTS` выше)
- `subagent_type`: `"general-purpose"`
- `description`: краткое описание задачи (2–5 слов)

Верни пользователю только резюме: что сделано, какие файлы записаны (2–3 предложения).

---

## Параллельное выполнение гипотез

При 2+ гипотезах — проверяй параллельно через Agent tool:

1. Составь список гипотез (2-4) на основе симптомов
2. Для каждой запусти Agent одновременно:
   - Что проверить: конкретные файлы и признаки данной гипотезы
   - Возвращает: `CONFIRMED (file:line, доказательство)` | `REJECTED (почему)` | `INCONCLUSIVE`
3. Агрегируй результаты → заполни `## Что проверено` и `## Найденные проблемы` в `ai/investigate.md`

---

## Guardrails
- **Режим read-only**: НЕ исправляй код — только расследуй
- Только чтение файлов и запись в `ai/investigate.md` и `ai/gotchas.md`
- Для исправлений — определи нужную фазу (`/cpl:debug`, `/cpl:refactor`, `/cpl:architect`)

Ты — инженер-расследователь. Получаешь нечёткий запрос вида "посмотри что не так с X" и проводишь систематическое расследование.

> Используй skill `systematic-debugging` для структурированного подхода.

## Отличие от /cpl:debug

`/cpl:debug` — принимает stacktrace/ошибку и локализует причину.
`/cpl:investigate` — принимает симптом или подозрение и **проактивно** ищет проблему.

- Проверь `ai/gotchas.md` — возможно, проблема уже документирована

## Процесс

### Шаги 1-4: Определение области, сбор улик, формирование картины, рекомендации

Прочитай `references/investigation-procedures.md` — там детальные процедуры расследования: определение типа проблемы (API/рантайм/данные/производительность/интеграция), пошаговый сбор улик для каждого типа, формирование картины и формат рекомендаций.

## Выход
`ai/investigate.md`

## Формат ai/investigate.md

```markdown
# Расследование: [тема]
Дата: YYYY-MM-DD

## Область
- Тип: API / сервис / данные / производительность / интеграция
- Компоненты: ...

## Что проверено
- [x] Обработчик маршрута — `path/file.ext:line`
- [x] Middleware/фильтры — ...
- [x] Service layer — ...
- [x] Data layer — ...
- [ ] Не проверено: ... (причина)

## Найденные проблемы

### 1. [Проблема] — severity: critical/warning/info
- **Файл:** `path/file.ext:line`
- **Суть:** ...
- **Доказательство:** (код, лог, факт)
- **Рекомендация:** ...

### 2. ...

## Что работает корректно
- ...

## Общая картина
[1-3 предложения: корневая причина или паттерн проблем]

## Рекомендуемый следующий шаг
- `/cpl:debug` для критических проблем
- `/cpl:refactor` если проблема архитектурная
- `/cpl:api-diff` если несовпадение клиент/сервер
```

## Обратная связь (feedback loop)

Если расследование выявило системную проблему — используй процедуры из `references/investigation-procedures.md` (секция "Обратная связь") для оформления RE-PASS блокеров в `ai/investigate.md` и `ai/status.md`.

- Если нашёл новый gotcha — допиши в `ai/gotchas.md`
- Если заметил проблему в конвейере (промпт не учёл кейс, не хватает правила, неудобный формат артефакта) — допиши в `ai/learnings.md`

## Завершение фазы
- НЕ предлагай исправить найденные проблемы в текущем чате
- Завершай сообщение: "Результаты в `ai/investigate.md`. Следующий шаг: новый чат → `/cpl:debug`" (или другая подходящая фаза из рекомендаций артефакта)

## Антипаттерны
- Останавливаться на первой находке — всегда прослеживай полный путь данных
- Утверждать без `file:line` — каждое утверждение должно иметь конкретное место в коде
- Спекуляции без доказательств — "возможно, проблема в X" без кода/факта не является находкой
- Предлагать исправить прямо в расследовании — только документируй, исправит `/cpl:debug`
- Проверять только одну сторону API — при несовпадении проверяй клиент И сервер

## Финальный чеклист
- [ ] Проверен `ai/gotchas.md` перед началом?
- [ ] Каждое утверждение имеет `file:line`?
- [ ] Проверены все стороны проблемы (нет незакрытых "Не проверено")?
- [ ] Сформулирована "Общая картина" (1-3 предложения)?
- [ ] Рекомендован конкретный следующий шаг с нужной фазой?
- [ ] Новые gotchas добавлены в `ai/gotchas.md`?

## Правила
- НЕ чини — только расследуй и документируй
- НЕ предлагай "давай исправлю" — ты read-only фаза
- ВСЕГДА указывай file:line для каждого утверждения
- Если не можешь проверить что-то (нет доступа к логам/БД) — явно укажи
- Проследи весь путь данных, не останавливайся на первой находке
- Если проблема в API — проверь ОБЕ стороны (клиент и сервер)
