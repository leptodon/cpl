# /cpl:new-project

**Recommended model: opus**

Ты — архитектор-инициализатор. Определяешь тип проекта, стек, и настраиваешь конвейер cpl под конкретный проект. Работает полностью автономно — не требует предварительного запуска init.sh.

## Задание
$ARGUMENTS

## Выполнение через Agent

**СТОП. Не выполняй задание — вызови Agent tool прямо сейчас.**
- `prompt`: весь текст этого скилла (включая `$ARGUMENTS` выше)
- `subagent_type`: `"general-purpose"`
- `description`: краткое описание задачи (2–5 слов)

Верни пользователю только резюме: что сделано, какие файлы записаны (2–3 предложения).

---

## Шаг 0: Bootstrap — создание рабочего пространства

### Определи режим размещения

Если аргумент содержит `inline` — все файлы создаются внутри проекта.
Иначе — используется sibling-режим (по умолчанию):

1. Создай **sibling-директорию** `<ProjectName>Ai` рядом с проектом:
   - Пример: проект `/path/MyApp` → AI-файлы в `/path/MyAppAi/`
2. Внутри `<ProjectName>Ai/` создай `ai/`, `ai/history/`, `CLAUDE.md`
3. В **директории проекта** создай **символические ссылки**:
   - `ai/` → `../<ProjectName>Ai/ai/`
   - `CLAUDE.md` → `../<ProjectName>Ai/CLAUDE.md`
4. Если ссылки уже существуют (файл/директория) — НЕ перезаписывай, сообщи пользователю
5. Добавь `ai` и `CLAUDE.md` в `.git/info/exclude` проекта (локальный gitignore, не коммитится)

### Создай артефакты

Создай все необходимые артефакты (пустые шаблоны) если не существуют (в `<ProjectName>Ai/` или inline):

### Обязательные артефакты (для всех проектов):
- `ai/context.md` — контекст проекта
- `ai/requirements.md` — требования
- `ai/architecture.md` — архитектура и ADR
- `ai/tasks.md` — план задач
- `ai/impl_notes.md` — заметки реализации
- `ai/review.md` — результаты ревью
- `ai/debug.md` — отладка
- `ai/status.md` — статус конвейера
- `ai/gotchas.md` — подводные камни
- `ai/learnings.md` — наблюдения о процессе
- `ai/working_regulation.md` — рабочий регламент
- `ai/handoff.md` — передача контекста

### Условные артефакты (по типу проекта):
- `ai/design.md` — если есть UI (mobile/frontend/fullstack)
- `ai/ui_spec.md` — если есть UI
- `ai/api_contracts.md` — если есть API (backend/fullstack)
- `ai/build_and_ci.md` — если есть сборка
- `ai/test_plan.md` — если планируются тесты
- `ai/investigate.md` — пустой шаблон для расследований

Каждый файл создавай с минимальным заголовком:
```markdown
# [Название]: [Имя проекта]
Дата: [сегодня YYYY-MM-DD]

[пустые секции по шаблону]
```

**НЕ перезаписывай существующие файлы.**

### Шаблон CLAUDE.md (если не существует):
```markdown
# [Имя проекта] — Claude Code Guidelines

## Проект
[Имя проекта]

## Профиль проекта
- **Тип**: [определить в шаге 2]
- **Стек**: [определить в шаге 2]

## Сборка
Сборка и запуск выполняются вручную через IDE/локальные процессы.
Claude не должен запускать сборку/рантайм команды без явного запроса.

## Конвейер cpl (Claude Pipeline)

**Принцип: не смешивать фазы в одном чате. Контекст между фазами фиксируется в `ai/*.md`.**

### Автороутинг

Если пользователь пишет запрос без явной команды — используй `/cpl:do` для определения подходящей фазы.

| Признаки запроса | Фаза |
|---|---|
| "посмотри", "разберись", путь к файлу | `/cpl:context` |
| "хочу фичу", "нужно чтобы" | `/cpl:product` |
| "как построить", "структура" | `/cpl:architect` |
| "напиши", "реализуй" + код | `/cpl:gen` или `/cpl:gen-server` |
| "проверь", "ревью" | `/cpl:code-review` |
| "ошибка", stacktrace | `/cpl:debug` |
| "что сделано", "прогресс" | `/cpl:progress` |
| "что дальше" | `/cpl:next` |

### Правила конвейера
1. НЕ генерить код без плана (`tasks.md`)
2. НЕ дебажить во время генерации
3. Каждый агент пишет ТОЛЬКО в свой артефакт
4. CodeGen: 1 таск → 1 коммит (1–3 максимум)
5. Каждая фаза пишет наблюдения в `ai/learnings.md`
6. Перед генерацией — читать `ai/gotchas.md`
```

## Шаг 1–3: Разведка, профиль проекта, выбор фаз

Прочитай `references/project-profiles.md` — там инструкции по разведке проекта, шаблон профиля, таблицы для определения типа проекта, языка, фреймворков, БД, DI, CI, и правила выбора фаз конвейера.

## Шаг 4: Наполни артефакты стек-специфичными секциями

После подтверждения профиля обнови созданные в Шаге 0 артефакты — добавь релевантные секции для определённого стека. Пустые секции, нерелевантные для стека — удали.

## Шаг 5: Инжекция паттернов в CLAUDE.md

Прочитай `references/project-profiles.md` — секция "Инжекция стек-специфичных паттернов". Добавь в `CLAUDE.md` проекта:
1. Таблицу фаз конвейера (команда, модель, читает, пишет)
2. Секцию "Паттерны проекта" с примерами кода для определённого стека
3. Правила конвейера

**Важно**: Добавляй паттерны ТОЛЬКО для определённого стека.

## Шаг 6: Настрой ai/status.md

```markdown
# Статус: [Проект]
Дата: YYYY-MM-DD

## Профиль
- Тип: [из шага 2]
- Стек: [из шага 2]

## Конфигурация
- verify_cmd: [авто-определение, см. ниже]
- tdd: false

## Фаза
- Текущая: **инициализация**
- Следующая: `/cpl:context` или `/cpl:kickoff`

## Прогресс по артефактам
- [ ] context.md
- [ ] requirements.md
- [ ] ...
```

### Auto-detect verify_cmd:
- `build.gradle.kts` / `build.gradle` → `./gradlew build`
- `package.json` с "build" скриптом → `npm run build`
- `Cargo.toml` → `cargo check`
- `pyproject.toml` → `python -m py_compile`
- `go.mod` → `go build ./...`
- `pom.xml` → `mvn compile`
- Если неоднозначно — спроси пользователя

## Шаг 7: Рекомендация

Выведи:
```
## Готово! Проект настроен.

Профиль: [тип] / [стек]
Создано артефактов: N
verify_cmd: [команда]

Следующий шаг:
- `/cpl:context` — собрать контекст из существующего кода/доков
- `/cpl:kickoff [описание фичи]` — быстрый старт для конкретной задачи
```

## Правила
- НЕ пиши production-код
- НЕ угадывай стек — читай файлы или спрашивай
- НЕ создавай артефакты для ненужных фаз
- Секции в артефактах ДОЛЖНЫ соответствовать стеку проекта
- Минимализм: лучше добавить фазу позже, чем создать лишнюю
- Если проект уже зрелый — предложи `/cpl:context` для сбора текущего состояния
